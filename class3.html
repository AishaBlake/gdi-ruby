Objects

An object is the building block of a program.

There will be many objects all working together to get something done.

An object can represent something in real life like a Coffee or something you can't see like a Birthday or an action like the Path you walk around Green Lake.

Objects are capable of doing things. They also have properties like color or size.


Object: Bike
What can it do?
- Exist
- Go
- Brake
- Pop a Wheelie

What are some properties of your Bike?
- Brand
- Size
- Wheel Size
- Number of Speeds
- Material

Object Class

Objects can be grouped by Class.

Classes define a template for objects. They describe what a certain type of object can do and what properties they have in common.


class Bike
end

bike = Bike.new
bike.class
=> Bike

Methods

A method is a series of statements. Sometimes methods are called functions.

Methods are often used to communicate between objects.

For example, a Bike object might receive a message to `pop_wheelie`.


class Bike

  def pop_wheelie
    puts "popped wheelie!!"
  end

end

bike = Bike.new
bike.pop_wheelie
=> "popped wheelie!!"

Built in Methods

Some methods have already been defined on your object.

For example, we can call `new` on Bike to create a new instance from our Bike template


bike = Bike.new
bike.methods

bike.methods - Object.methods

Object Methods

Methods have access to the properties of their objects.


class Bike
  def set_brand
    @brand = "Fuji"
  end

  def get_brand
    puts "The brand of the bike is: #{@brand}"
  end
end

bike = Bike.new
bike.set_brand
bike.get_brand
=> "The brand of the bike is: Fuji"

Method Input

Methods can do things with data given to them.


class Bike
  def set_brand
    @brand = "Fuji"
  end

  def get_brand
    puts "The brand of the bike is: #{@brand}"
  end

  def go_speed(speed)
    if speed < 15
      puts "Safety first."
    else
      puts "Speed racer!"
    end
  end
end

Methods & Variables

Local variables are only available in the methods where they were defined.

Instance variables are available to all the methods in the class.


class Bike
  def set_brand
    @brand = "Fuji"
  end

  def get_brand
    puts "The brand of the bike is: #{@brand}"
  end

  def go_speed(speed)
    if speed < 15
      puts "Safety first."
    else
      puts "Speed racer!"
    end
  end

  def get_speed
    puts speed
  end
end

bike = Bike.new
bike.get_speed
=> NameError: undefined local variable or method `speed'

Methods & Return Values

Every Ruby method returns something.

Usually, it's the last statement in the method.

If you don't want to return the last statement, you'll use a reserved word return

Notice that the method puts returns nil


def return_early
  return "This is what you get!"
  1*2
end

return_early
=> "This is what you get!"

puts "Hi!"
"Hi"
=> nil

Objects Working Together

Object Oriented Programming is one of many programming approaches: procedural, event-driven, functional, declarative...

All this means is that programs are organized around objects (data & methods).

You might choose OOP because Objects make things easier to understand and act as a natural way to modularize code. When a large code base is composed of many small Objects, it is easier to maintain and change.

OOP & Easy Understanding

Objects often take on names from their domain which makes them easy to conceptualize.

For example, without seeing my code, you can probably guess what my Bike can do (or not do)

OOP & Maintenance

One thing that makes code easier to maintain is Encapsulation which is one of the fundamentals of OOP.

Encapsulation is a nice way of saying 'put all my properties/behaviors in my capsule where other objects can't touch them'.

By assigning certain responsibilities to objects, code is more purposeful and less likely to be changed by accident.

OOP & Complexity

As code becomes more complex, we as programmers need to think about which parts of the object are relevant to others.

Abstraction is used to say only what is needed.

For example, your Bike might have tire_pressure and spoke_count properties that help indicate whether the bike is being maintained. Other objects may not need to know the actual maintenance details and you might be able to abstract this data out into a more relevant method `maintained?`.


class Bike
  def perform_maintenance
    @tires_filled = YES
    @cleaned = YES
    @bolts_tightened = YES
  end

  def maintained?
    @tires_filled & @cleaned & @bolts_tightened
  end
end

OOP & Borrowing

Just as you inherited traits from your mom, your objects may inherit traits from other objects.

Inheritance is when you use on class as the basis for another.

You might use Inheritance if two objects have the same properties and methods. This way you don't need to write the same thing twice.


 class RoadBike < Bike
 end

 road_bike = RoadBike.new
 road_bike.perform_maintenance

OOP & Ducks

Similar to inherited traits and behaviors, there are also polymorphic traits and behaviors.

Polymorphic means that something can take on many forms.

Within OOP, this relates to different objects being able to respond to the same message in different ways.


class Fixie
 def gears
   1
 end
end

class RoadBike
 def gears
   21
 end
end

[RoadBike.new, Fixie.new].each {|bike| puts bike.gears}
